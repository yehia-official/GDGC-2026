Build Your Personal Bio Page
HTML basics · Structure · Tags · Text · Lists
▼
Beginner
⏱ 45–90 min
📁 1 HTML file
🎬 The Scenario You're applying for a junior developer internship. The company asks you to send a simple HTML-only bio page about yourself before the interview. No CSS, no JavaScript — just clean, well-structured HTML. They want to see if you understand how to structure content meaningfully.
What you need to build
  • 1
    A proper HTML document with a meaningful <title> that shows in the browser tab.
  • 2
    A clear heading hierarchy — your name as the main title, and logical sub-headings for each section of your life.
  • 3
    A short paragraph introducing yourself. Use text emphasis where it genuinely makes sense — don't bold random words.
  • 4
    Two lists: one for your skills (order doesn't matter), and one for a goal you want to achieve — written as steps in the right order.
  • 5
    A "favorite quote" section that uses the correct semantic element for quotes, with a credit to whoever said it.
  • 6
    A thematic divider between your intro and your skills sections.
Self-check — tick what you've done
My HTML file opens without errors in the browser
There is exactly one <h1> on the page — my name
I used <strong> or <em> because the content is actually important — not just for looks
My skills list is <ul> and my steps list is <ol> — and I know why they're different
The quote uses the right semantic HTML element, not just italic text
If I view the page source, another developer can understand the structure without seeing it rendered
Forbidden
Any CSS — not even style="" inline attributes. Raw HTML only.
Using heading tags just to make text bigger. If you write <h2>, that section must actually be a second-level section.
Using <br> to create spacing between sections. That's what block elements are for.
Stuck? Expand hints
Think about which HTML element was literally designed for quotations. It's not <p> and it's not <em>.
The difference between <ul> and <ol> is about meaning, not appearance. Ask yourself: does the order of these items matter in real life?
A good heading hierarchy looks like a table of contents. Read your headings out loud in order — do they make sense as an outline?
The <title> tag goes inside <head> — it's what shows in the browser tab and in Google search results.

A Product Comparison Page
Links · Images · Attributes · Forms · Semantic HTML · Tables
▼
Intermediate
⏱ 60–120 min
📁 1 HTML file
🎬 The Scenario A friend is starting a small online store selling 3 types of headphones. They asked you to build a simple comparison page that shows the products side by side, lets visitors contact them, and links to product pages. They have no budget yet — just a plain HTML page for now.
What you need to build
  • 1
    Use proper semantic HTML for the page layout: the site logo/name in a <header>, navigation links in a <nav>, the product area inside <main>, and copyright info in a <footer>.
  • 2
    A comparison table with at least 3 products and 4 attributes (e.g., price, battery life, color, rating). Make sure the table header row is semantically different from the data rows.
  • 3
    Each product name in the table should link to an external page (any URL). One of them should open in a new tab, the others should not. Be intentional about which one and why.
  • 4
    At least one product image with a meaningful alt text. If the image can't load, the alt text should still tell the user what they're missing.
  • 5
    A contact form at the bottom with: name, email, a dropdown to select which product they're asking about, a message textarea, and a submit button. All fields should be properly labeled.
  • 6
    Use the id attribute on the contact form section, and put a "Contact Us" link in the <nav> that jumps to it when clicked.
Self-check — tick what you've done
I can explain why each semantic element I used (header, nav, main, footer) is the right choice for that content
My table has <thead> and <tbody> — and I know the difference between <th> and <td>
Every <label> in my form is linked to its input with matching for="" and id=""
My image alt text describes the image content — not just "image" or "photo"
The nav "Contact Us" link actually jumps to the form when clicked
I made a deliberate choice about which link opens in a new tab — and I can explain why
Forbidden
Using <div> where a semantic element exists. If there's a semantic element that fits, use it.
Making every link target="_blank". Only external links that would take the user away from the current context warrant a new tab.
Writing alt="image" or alt="product photo". The alt text should work as a replacement for the image if it doesn't load.
Stuck? Expand hints
An anchor link to a section on the same page looks like href="#section-id", and the target element needs that id on it.
The <select> element creates a dropdown. Each option inside it needs a value attribute and visible text between the tags.
Think about when you use target="_blank" in real life. If you're linking to the store's own pages — you don't want to open a new tab. If you're linking to an external review site, a new tab makes sense.
A table header cell <th> is for labels (Product Name, Price, etc). A data cell <td> is for the actual data. Screen readers use this distinction to announce column names to blind users.

A News Article Page
Multimedia · Entities · Meta Tags · Accessibility · Form Validation
▼
Intermediate+
⏱ 75–120 min
📁 1 HTML file
🎬 The Scenario You're building an article page for a small tech news website. The article is about a fictional AI tool — you'll write the content yourself (keep it short, 2–3 paragraphs). The editor wants the page to look good when shared on WhatsApp and LinkedIn, load fast, and be accessible to readers who use screen readers.
What you need to build
  • 1
    A complete <head> section with: page title, UTF-8 charset, viewport meta tag, a description meta tag (max 160 characters), and at least 3 Open Graph tags so the article looks good when shared on social media.
  • 2
    The article content wrapped in the right semantic element — because it's self-contained content that could exist independently from this website. Include a publication date using the element designed for dates/times.
  • 3
    An embedded YouTube video related to your topic (use any video). Make sure it has a title attribute describing the video for screen readers, and loads lazily.
  • 4
    At least 2 HTML entities used correctly in the article text — for example, the copyright symbol, an em dash, a trademark symbol, or any special character that belongs in the content.
  • 5
    A "newsletter signup" form at the end of the article. It needs: an email field that validates format, a name field that's required and has a minimum of 2 characters, and a submit button. All fields must be properly labeled.
  • 6
    At least one image with proper accessibility: meaningful alt text, and the loading="lazy" attribute since it's below the fold.
Self-check — tick what you've done
If I share this page's URL on WhatsApp, it would show a proper preview (title, description, image) — not just a raw URL
I used the right semantic element for the article and the publication date — and I know why
The iframe has a title attribute that would make sense to a blind user who can't see the video
My entities are used because the character genuinely belongs there — not just pasted in randomly
If I try to submit the form with an invalid email or empty name, the browser shows a validation error without JavaScript
My image would still be understandable to someone who can't see it, based on the alt text alone
Forbidden
Skipping the Open Graph tags and hoping it "looks fine" when shared. Check it properly.
Using <b> or <i> — these are presentational. Use <strong> and <em> for meaning, or CSS for pure styling.
Writing alt="image of article" or leaving alt empty on a content image. If the image is content, describe it.
Adding JavaScript to handle form validation. All validation in this task must be HTML attributes only.
Stuck? Expand hints
Open Graph tags use property="" not name="". The ones you need: og:title, og:description, og:image.
The element designed for dates and times is <time>. It has a datetime attribute for the machine-readable format: datetime="2025-01-15".
To embed a YouTube video, change the URL from youtube.com/watch?v=ID to youtube.com/embed/ID and put it in an <iframe>.
The em dash entity is &mdash;. The copyright is &copy;. Use them where they genuinely belong in the prose.
HTML form validation for minimum length is minlength="2". For email format validation, just using type="email" is enough.

Build a Complete HTML Website
Global Attributes · Responsive Images · Everything
▼
Advanced
⏱ 2–4 hours
📁 1 HTML file
🏆 Capstone
🎬 The Scenario A friend who's a freelance photographer wants a simple online portfolio. They've never had a website. You're going to build the full HTML structure for their site — no CSS yet, just clean, complete, production-ready HTML that any developer could later add styles to. This is the real thing. Take your time.
What you need to build
  • 1
    Complete head section: charset, viewport, page title, description, author, Open Graph tags (title, description, image, type), Twitter card, and a canonical link. Every tag must have real, meaningful content — not placeholder text.
  • 2
    Semantic structure: A <header> with a site name and nav. A <main> containing at least 3 <section>s: About, Portfolio, and Contact. A <footer> with copyright (using the right entity) and social links.
  • 3
    Portfolio gallery: At least 4 photos using responsive images — the <picture> element with at least a WebP source and a JPEG fallback. Every image must have loading="lazy", decoding="async", and a genuinely descriptive alt text.
  • 4
    Global attributes: Use data-* attributes on the portfolio images to store the photo category and year. Add lang to any non-English text on the page. Use contenteditable on the "tagline" so the visitor can type their own version right on the page.
  • 5
    Fully accessible contact form: Name (required, minlength 2), email (required, email type), project type dropdown, budget range with a <datalist> for suggestions, message (required), and a checkbox for newsletter consent. Every field must have a linked label. Group the personal info fields in a <fieldset>.
  • 6
    Accessibility pass: Icon-only buttons must have aria-label. Decorative elements (if any) must have aria-hidden="true". The page must have a logical heading hierarchy. The <html> tag must have the correct lang attribute.
Self-check — tick what you've done
I opened the page and read all headings out loud — they make a logical outline of the site
I disabled images in my browser — the alt texts alone still tell the full story of the portfolio
I tried navigating the form using only the keyboard (Tab key) — every field is reachable in logical order
I tried sharing the page URL in a WhatsApp message — a proper card appeared with title, description, and image
I can open browser DevTools, click any portfolio image, and see its data-category and data-year in the attributes panel
I tried submitting the form empty and with a bad email — the browser caught both errors without any JavaScript
I validated the HTML at validator.w3.org — zero errors (warnings are okay)
I looked at the page source and I'm genuinely proud of how clean and readable the structure is
Forbidden
Using alt="" on any portfolio photo. Every photo in a photography portfolio is content — it must be described.
Using <div> where a semantic element clearly fits. After 17 modules, you know better.
Faking the Open Graph tags with placeholder text. Write real content as if this were a real photographer's website.
Using JavaScript for any validation or interaction in this task. Everything must be native HTML.
Copy-pasting from the masterclass examples. You can reference them, but write every line yourself.
Bonus challenges (optional)
  • +
    Add a <details> and <summary> element for an FAQ section — research these yourself, they weren't in the course.
  • +
    Validate your page with the W3C HTML validator. Fix every error you find and understand why each one was flagged.
  • +
    Run the page through a free accessibility checker (like WAVE or axe DevTools) and fix every critical issue it finds.
Stuck? Expand hints
For data-* attributes: data-category="landscape" and data-year="2024" — then in JS (later) you access it via element.dataset.category.
The <datalist> element gives suggestions to an <input>. Give both the same id/list pair: <input list="budgets"> then <datalist id="budgets">.
For <picture> with free images: use picsum.photos as your source. WebP and JPEG can point to the same URL — the important thing is that you're using the correct HTML structure.
To check Open Graph tags without publishing, paste your HTML into opengraph.xyz or use a browser extension like "Open Graph Preview".
The copyright entity is &copy; — use it in your footer like: &copy; 2025 Photographer Name.

A Note on Using AI

Yes, you can ask AI to explain something you don't understand. That's fine — it's like asking a teacher. But if you paste a task into an AI and copy the output, you've learned absolutely nothing. The struggle is the learning. Every bug you fix yourself sticks. Every bug AI fixes for you disappears.

The goal isn't a working file. The goal is a working brain. 🧠
